16.1 The Full Decision Tree

When starting a new AI automation, work through these questions in order:

  • What is the trigger? Schedule, event, webhook, manual, or reactive pull?
  • What data does it process? Is any of it sensitive/PII/regulated?
  • Does it need to be stateful? (Conversation history, prior context, session memory)
  • What quality is required? (Best model vs fast/cheap model vs local)
  • What is the expected volume? Calculate monthly cost at that volume.
  • What are the failure modes? What should happen when the API is down?
  • What is the output? File, notification, API call, database write, git commit?
  • How will you know if it breaks? Logs, metrics, alerts, eval suite?

Closing: The Automator's Discipline

Every tool in this manual solves a real problem. None of them are necessary in isolation. The art is in selecting the minimum combination that solves your specific problem without creating new complexity.

The Akasha system is a good example of this discipline in practice. It does not use LangGraph because a simple JavaScript routing function suffices. It does not use Airflow because n8n covers the orchestration. It does not use CrewAI because a single model with clear instructions outperforms three agents debating. It uses Tasker because nothing else has Android system-level access. It uses Redis because cross-process session state needs a real store. It uses the outbox pattern because mobile network reliability demands it.

Every addition to a system adds maintenance burden, failure surface, and cognitive load. The best automators are not those who use the most tools — they are those who know exactly when each tool earns its place.

Build incrementally. Measure everything. Version your contracts. Test before you deploy. Know your costs. Know your failure modes.

That is the whole word.

— The Compiler
