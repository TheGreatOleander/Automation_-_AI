PART XXIV

The Agent Triad

Input/output contracts for autonomous Code Volleyball — the inner loop without the human.

24.1 Overview

Part XXII described Code Volleyball as a human-orchestrated relay: you zip the project, write the next prompt, pass it to a model, review the output, zip again. The Prompt Ledger is the mechanism that makes each handoff coherent. That pattern works. It also has a ceiling — the human in the loop is both the quality gate and the bottleneck.

The Agent Triad is what Code Volleyball becomes when you remove the human from the inner loop. Three agents cooperate to move from structural insight to built artifact without requiring human translation between them. The Ledger becomes live state. The relay becomes autonomous. The human moves from relay runner to checkpoint reviewer.

Relationship to Part XXII: The Triad does not replace the Prompt Ledger — it operationalises it. Logistics Guy's blueprint is the Ledger's Current Build State. The gap report is the Open Issues list. The task emitted to Builder is the Next Pass Prompt. The same structure, running without a human triggering each step.

You review outputs at natural checkpoints — not at every step. The loop runs until a blocker persists, scope diverges, or a completion reveals a gap large enough to warrant human judgment.

24.2 Agent 1 — Idea Guy

The Mendeleev role. Reads the shape of existing structure. Names what is absent because the structure demands it — not because it would be useful or interesting, but because the system is incomplete without it.

Core mandate: Detect structural holes — configurations that must exist for the system to be complete — and produce exactly one, precisely specified gap report per invocation. Nothing more.

Input Contract

FieldDescription
current_structureThe existing system: entities, relations, constraints, invariants
known_holesList of gaps already identified — checked before output to prevent duplication
focus_domainOptional: constrain search to a specific subsystem

Output Contract

Exactly one gap per invocation. Output must be a structured record — not prose.

FieldDescription
gap_idUnique identifier (e.g. GAP-007)
structural_demandThe specific invariant or constraint violated by this absence
evidenceWhich existing elements imply the missing one — cited explicitly
shape_of_missingWhat the absent element must look like — minimum necessary specification
priority_signalWhy this hole creates the most tension in the current structure
out_of_scopeWhat this gap report does NOT address — prevents scope creep downstream

Prohibitions

MUST NOTMUST
Produce more than one gap per invocationIdentify exactly one gap and stop
Generate ideas without structural justificationEvery output must cite a violated invariant
Speculate about implementationshape_of_missing is a spec, not a build plan
Restate previously identified gapsCheck known_holes before output
Use vague language ("we might need...")Be precise: "constraint X is unsatisfied because Y is absent"
Failure mode this prevents: Idea Guy going into brainstorm mode — producing overlapping, ambiguous, or enthusiasm-driven ideas that have no structural grounding and cannot be acted upon by Builder.

24.3 Agent 2 — Logistics Guy

The Foreman. Speaks both languages. Translates structural gap into actionable build sequence and keeps the map updated as understanding evolves. This is the pivot agent — every message in the system passes through or is acknowledged by Logistics Guy.

Core mandate: Receive a gap report from Idea Guy. Decompose it into an ordered dependency tree of build tasks. Package tasks for Builder. Update the blueprint after each Builder completion. Surface blockers before they halt work.

Input Contract — From Idea Guy

FieldDescription
gap_reportFull structured gap record from Idea Guy
current_blueprintExisting dependency map of all prior and in-progress work
builder_capabilitiesWhat Builder can currently execute: languages, tools, APIs

Input Contract — From Builder

FieldDescription
task_idWhich task was completed or blocked
statusCOMPLETE | BLOCKED | PARTIAL
artifactWhat was produced: file path, schema, API endpoint, etc.
blocker_descriptionIf BLOCKED: precise description of what is missing

Output Contract — To Builder

FieldDescription
task_idUnique task identifier (e.g. TASK-012)
depends_onList of task_ids that must be complete before this one
descriptionWhat to build — precise enough to execute without clarification
acceptance_criteriaHow to verify the task is done
context_refsRelevant prior artifacts or constraints to incorporate
expected_outputWhat artifact should exist when done

Output Contract — Blueprint Update

After each Builder completion, Logistics Guy emits an updated blueprint showing: completed nodes, open nodes, in-progress nodes, and any newly revealed dependencies. The blueprint is the ground truth of system state — equivalent to the Ledger's Current Build State in the manual relay workflow.

Prohibitions

MUST NOTMUST
Pass ambiguous tasks to BuilderEvery task must be self-contained and executable
Skip dependency orderingNo task goes to Builder before its dependencies complete
Invent requirements not in gap reportDecompose only what Idea Guy specified
Suppress blockersBlockers must surface immediately with precise description
Let blueprint go staleUpdate after every Builder response, no exceptions
Failure mode this prevents: Builder receiving tasks that are under-specified, out-of-order, or in conflict with existing structure — causing silent failures or wasted work that requires human intervention to untangle.

24.4 Agent 3 — Builder

Deterministic executor. Takes a well-specified task and produces an artifact. Does not interpret intent — only executes specification. The Builder's discipline is the discipline of not deciding: when something is unclear, the answer is always BLOCKED, never a creative interpretation.

Core mandate: Execute exactly what is specified in the task. Produce the artifact described in expected_output. Verify against acceptance_criteria. Report status with precision. Never extrapolate beyond the task boundary.

Input Contract

FieldDescription
task_idTask identifier from Logistics Guy
descriptionWhat to build
acceptance_criteriaDone-when conditions
context_refsArtifacts and constraints to incorporate
expected_outputWhat artifact to produce

Output Contract

FieldDescription
task_idEcho the task_id — confirms correct task was executed
statusCOMPLETE | BLOCKED | PARTIAL
artifactLocation or description of produced artifact
acceptance_verifiedConfirmation that each criterion was checked — list them explicitly
blocker_descriptionIf BLOCKED: what is missing — surgical, not general
out_of_scope_notedAnything observed but intentionally not built — task boundary preserved

Prohibitions

MUST NOTMUST
Extend scope beyond task specIf it is not in the task, do not build it
Assume missing contextIf context is insufficient, report BLOCKED immediately
Silently skip acceptance criteriaAll criteria must be explicitly verified in output
Produce non-reproducible artifactsDeterminism required
Interpret ambiguous specsAmbiguity = BLOCKED, not a creative decision
Failure mode this prevents: Builder silently over-building, under-building, or making interpretive decisions that invalidate the structural reasoning that originated the task — causing the gap to remain open despite apparent completion.

24.5 Handoff Rules

Idea Guy → Logistics Guy

Idea Guy emits exactly one gap report. Logistics Guy acknowledges receipt and returns either a decomposed task tree or a request for clarification on exactly one ambiguous field. No gap report is acted upon until Logistics Guy has confirmed it is buildable.

Logistics Guy → Builder

Logistics Guy emits exactly one task at a time. Builder does not receive task N+1 until task N is COMPLETE. If task N is BLOCKED, Logistics Guy resolves the blocker — potentially requesting a new gap from Idea Guy — before reissuing.

Builder → Logistics Guy

Every Builder output triggers a blueprint update. Logistics Guy either issues the next task or escalates to Idea Guy if a completion reveals a new structural hole.

24.6 Escalation to Human

The Triad runs autonomously until one of four conditions is met. Each condition requires a human decision before the loop continues — not a retry, not an inference, a decision.

ConditionAction
Blocker persists after 2 Logistics attempts Surface to human with precise description of the gap. Do not attempt a third resolution.
New gap discovered mid-build changes scope significantly Pause and surface to human before continuing. The original gap report is now stale.
Blueprint diverges from original gap report Surface delta to human for confirmation. Do not silently absorb scope drift.
Acceptance criteria cannot be verified Surface to human — do not auto-pass. An unverified criterion is a failed criterion.

24.7 Mapping to the Prompt Ledger

For teams running both the manual relay (Part XXII) and the autonomous Triad, the two systems share the same underlying structure. The Ledger is the Triad's state rendered for human readability.

Triad ConceptLedger Equivalent (Part XXII)
gap_reportOpen Issues entry
blueprintCurrent Build State
task emitted to BuilderNext Pass Prompt
Builder completionCompleted in This Pass
escalation to humanHuman review checkpoint
known_holesArchitecture Decisions Log (decisions already made)

When the Triad escalates to human, the human's natural response is to write a Ledger entry — updating the blueprint, documenting the decision, and writing the next prompt. The two patterns are designed to compose, not compete.

The practical implication: You can start a build with the manual Ledger relay, hand it to the autonomous Triad when the structure is stable enough, and resume manual control at any escalation point. The state is always legible because the blueprint and the Ledger are the same document at different levels of automation.
