PART XI

Security, Privacy, and Compliance

11.1 The AI-Specific Threat Model

AI automation introduces three security threats that conventional software does not have: prompt injection (malicious content in inputs manipulates model behaviour), data exfiltration via model outputs (sensitive data leaks through generated text), and supply chain attacks via model weights or libraries.

Prompt Injection at Scale

When automation processes external content — emails, web pages, user-generated text — any of that content could contain injection attempts. At scale, some will succeed. The defence is architectural: the model's outputs must be validated before they become actions, and actions must have explicit scope limits.

11.2 PII and Data Privacy

When user data flows through third-party AI APIs, it is subject to the API provider's data processing terms. For consumer APIs (OpenAI, Anthropic), data is generally not used for training (for API users), but it does pass through third-party servers. Understand what data you are sending before building.

PII Detection and Redaction

Local Processing for Sensitive Data

The most certain PII protection: do not send sensitive data to external APIs at all. Use a local model (Ollama) for any workflow that processes health data, financial data, legal documents, or any data subject to regulatory requirements (GDPR, HIPAA, SOC 2).

11.3 Compliance Checklist

Before deploying an AI automation system that processes data about people:

  • Identify all data categories flowing through the system (personal, health, financial, behavioural)
  • Map each data category to applicable regulations (GDPR, HIPAA, CCPA, SOC 2)
  • For each third-party API call, verify the provider's data processing agreement covers your use case
  • Implement data minimisation: send only the fields the AI actually needs
  • Implement retention limits: automated deletion of logs and outputs after the required period
  • Document the processing purpose in a data processing record
  • Test your PII detection on representative samples from your actual data
  • For GDPR: implement a subject access request handler that can find and export all data for a given person

11.4 Secrets Management at Scale

For systems beyond a single developer: environment variables in .env files are not secure enough. Use a secrets manager.
